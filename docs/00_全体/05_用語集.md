# 用語集

---

## 概要

KisoFrameworkおよび関連プロジェクトで使用される主要な用語・略語・技術用語の定義をまとめる。  
関係者間の共通認識形成やドキュメント参照時の補助を目的とする。

---

## 用語一覧

|----------------------|----------------------------------------------------------------------------------------|
| アーキテクチャ       | システム全体の構造や設計思想、構成要素の関係性を定める設計指針。                       |
| API                  | Application Programming Interface。システム間やモジュール間の連携インターフェース。     |
| APIファースト        | API設計を最優先し、システム開発を進める手法。                                          |
| CI/CD                | 継続的インテグレーション／継続的デリバリー・デプロイメント。自動化された開発・運用手法。|

| コミュニティ主導     | 社内外の多様なエンジニアが自発的に参加し、運営・改善を進める方針。                     |
| シングルサインオン   | 一度の認証で複数システムにアクセスできる仕組み。                                       |

| テストカバレッジ     | テストがソースコードのどの程度を網羅しているかを示す指標。                             |
| ドキュメント         | 設計書・仕様書・手順書など、開発・運用に関する文書全般。                               |
| ナレッジベース       | 知見やFAQ、事例などを体系的に蓄積・共有する仕組み。                                    |
| バックエンド         | サーバー側の処理やデータ管理を担うシステム部分。                                       |

| プロジェクトマネージャー（PM） | プロジェクト全体の進捗・課題・リソース管理を担う責任者。                       |

| モジュール化         | 機能を独立した部品として分割・整理する設計手法。                                       |
| ユースケース         | システム利用者の操作や利用シナリオを記述したもの。                                     |

| ワークフロー         | 一連の作業や処理の流れを定義したもの。                                                 |
| IaC                  | Infrastructure as Code。インフラ構成をコードで管理・自動化する手法。                   |
| OAuth2.0             | 標準的な認可プロトコル。外部サービス連携などで利用。                                   |

| REST                 | Web API設計の代表的なアーキテクチャスタイル。                                         |

| GraphQL              | 柔軟なデータ取得を可能にするAPIクエリ言語。                                            |
| Swagger/OpenAPI      | API仕様を記述・共有するための標準フォーマット。                                        |

| CIツール             | 継続的インテグレーションを実現するための自動化ツール（例：GitHub Actions）。           |
| 静的解析             | ソースコードを実行せずに品質やバグを検出する手法。                                     |
| デプロイ             | 開発したシステムを本番環境等に配置・公開する作業。                                     |
| ロギング             | システムの動作記録やエラー情報を記録する仕組み。                                       |
| 監査ログ             | 重要操作やアクセス履歴を記録し、監査・追跡を可能にするログ。                           |
| マイクロサービス     | 機能ごとに独立した小規模サービス群で構成するアーキテクチャ。                           |
| クラウドネイティブ   | クラウド環境を前提とした設計・開発手法。                                               |
| コンテナ             | アプリケーション実行環境をパッケージ化し、移植性・再現性を高める技術。                 |
| Kubernetes           | コンテナオーケストレーションのためのOSSプラットフォーム。                              |
| GitHub Actions       | GitHub上でCI/CDや自動化ワークフローを実現するサービス。                                |
| Storybook            | UIコンポーネントの開発・ドキュメント化ツール。                                         |
| Playwright/Cypress/Jest | フロントエンドの自動テストツール。                                                 |
| Figma/Adobe XD       | UI/UXデザイン・プロトタイピングツール。                                                |
| アセット             | システムやプロジェクトで利用される資産（画像・データ・コード等）。                     |
| イミュータブル       | 変更不可な性質。主にオブジェクトやインフラの状態管理で利用。                           |
| オーケストレーション | 複数のサービスやコンテナを自動的に管理・制御する仕組み。                               |
| オンプレミス         | 自社運用の物理サーバーやデータセンターでシステムを稼働させる形態。                     |
| クラスタリング       | 複数のサーバーやノードをまとめて一つのシステムとして運用する手法。                     |
| サービスディスカバリ | サービス間通信時に動的に接続先を解決する仕組み。                                       |
| シークレット管理     | パスワードやAPIキーなど機密情報を安全に管理する仕組み。                                 |
| ジョブ               | バッチ処理や定期実行される処理単位。                                                   |
| スケーラビリティ      | システムの拡張性。負荷や規模の増加に柔軟に対応できる能力。                             |
| ステートレス         | 状態を持たない設計。リクエストごとに独立して処理する。                                 |
| ステートフル         | 状態を保持する設計。セッションやデータを維持する必要がある場合に用いる。               |
| セッション管理       | ユーザーやプロセスごとの状態を管理する仕組み。                                         |
| タイムアウト         | 一定時間内に処理が完了しない場合に強制終了する仕組み。                                 |
| データバインディング | UIとデータモデルを自動的に同期する仕組み。                                             |
| データマイグレーション | データ構造や格納場所の変更・移行作業。                                               |
| デプロイメントパイプライン | ビルド・テスト・デプロイ等を自動化した一連の処理フロー。                        |
| トレーサビリティ      | 変更履歴や処理経路を追跡できる性質。                                                   |
| ノード               | クラスタや分散システムを構成する個々のサーバーやプロセス。                             |
| バックプレッシャー   | システムの過負荷時に処理速度を調整する仕組み。                                         |
| フォールトトレランス  | 障害発生時でもシステム全体が継続稼働できる設計。                                       |

| ポリグロット          | 複数のプログラミング言語や技術を組み合わせて利用すること。                             |
| マイグレーション      | システムやデータの移行作業全般。                                                       |
| ミドルウェア          | OSとアプリケーションの中間で動作し、共通機能を提供するソフトウェア。                   |
| メトリクス            | システムやアプリケーションの性能・状態を示す指標。                                     |
| モノレポ              | 複数プロジェクトを単一リポジトリで管理する手法。                                       |
| ランブック            | 障害対応や運用手順をまとめたドキュメント。                                             |
| レイテンシ            | 処理や通信にかかる遅延時間。                                                           |
| ロールアウト          | 新機能や変更を段階的に展開する運用手法。                                               |
| ロールバック          | 不具合発生時に以前の状態へ戻す操作。                                                   |
| ワークロード          | システムが処理する業務量や負荷。                                                       |
| 可観測性（オブザーバビリティ） | システム内部の状態を外部から把握できる性質。                                 |
| 冗長化                | 障害対策のために複数の構成要素を用意する設計。                                       |
| 負荷分散              | 複数のサーバーやサービスに処理を分散させる仕組み。                                     |
| 無停止リリース        | サービスを停止せずに新バージョンをリリースする手法。                                   |
| 有効化フラグ（Feature Flag） | 特定機能の有効・無効を動的に切り替える仕組み。                              |
| 横断的関心事（クロスカッティング） | 認証・ロギングなど複数箇所で共通して必要となる機能。                     |
| 継続的改善（Kaizen）  | 小さな改善を積み重ねて品質や効率を高める活動。                                       |

| 非機能要件            | 性能・信頼性・セキュリティなど機能以外の要件。                                         |
| サーバーレス         | サーバー管理を意識せずにアプリケーションを実行できるクラウドサービスの形態。           |
| マイクロフロントエンド | フロントエンドを複数の独立したチーム・技術で分割・開発・統合するアーキテクチャ。     |
| オブザーバビリティ   | システムの内部状態を外部から観測・分析できる性質。                                     |
| ブルーグリーンデプロイメント | 新旧2つの環境を切り替えて無停止リリースを実現する手法。                       |
| カナリアリリース     | 一部ユーザーに限定して新バージョンを段階的に展開する運用手法。                         |
| SLA/SLO/SLI          | サービスレベル契約/目標/指標。可用性や性能などサービス品質を定量化する基準。           |
| レトロスペクティブ   | 振り返り。開発サイクルごとに課題や改善点を議論するミーティング。                       |
| DRY                  | Don't Repeat Yourself。重複を避ける設計・実装原則。                                   |
| KISS                 | Keep It Simple, Stupid。シンプルな設計を重視する原則。                                 |
| YAGNI                | You Aren't Gonna Need It。不要な機能を実装しない原則。                                 |
| BFF                  | Backend For Frontend。フロントエンドごとに最適化したAPI層。                            |
| ペアプログラミング   | 2人1組で協力してコーディングを行う開発手法。                                           |
| モブプログラミング   | 複数人で1台のPCを使い共同で開発する手法。                                               |
| スクラム             | アジャイル開発手法の一つ。短いサイクルで開発・改善を繰り返す。                         |
| スプリント           | スクラムにおける短期間の開発サイクル。                                                 |
| プルリクエスト       | 変更内容のレビュー・マージを依頼する仕組み（GitHub等）。                               |
| イシュートラッキング | バグや要望など課題管理を行う仕組み。                                                   |
| シフトレフト         | 品質・セキュリティ対策を開発工程の早い段階で実施する考え方。                           |
| シフトライト         | 運用・監視・フィードバックを重視し、運用段階での改善を行う考え方。                     |
| フィーチャートグル   | 機能ごとに開発・リリースを管理するブランチ戦略。                                       |
| オートスケーリング   | 負荷に応じて自動的にリソースを増減させる仕組み。                                       |
| サービスメッシュ     | サービス間通信の制御・観測・セキュリティを担うインフラ層。                             |

| ゼロトラスト         | すべての通信・アクセスを信頼せず検証するセキュリティモデル。                           |
| SAST/DAST            | 静的/動的アプリケーションセキュリティテスト。                                          |
| SBOM                 | Software Bill of Materials。ソフトウェア構成部品表。                                   |
| SSO                  | Single Sign-On。シングルサインオン。                                                   |
| RBAC                 | Role-Based Access Control。ロールベースアクセス制御。                                  |
| ABテスト             | 複数パターンを比較し効果を検証するテスト手法。                                         |
| ユニットテスト       | 単体テスト。最小単位の機能ごとに動作を検証するテスト。                                 |
| E2Eテスト            | End-to-Endテスト。システム全体の動作を検証するテスト。                                 |

| MCP（Model Context Protocol） | Googleが提唱したAIモデルとアプリケーション間の標準プロトコル。モデルの入出力やツール呼び出し、メタデータなどを共通形式でやり取りでき、異なるAIサービス間の互換性・拡張性を高める。 |
